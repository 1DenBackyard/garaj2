# PROJECT_MEMORY

## 1) Цель продукта
NeGaraj — сайт и мини-CRM автосервиса (фокус на BMW), где клиент может:
- увидеть услуги и ориентиры по стоимости работ,
- собрать набор работ в калькуляторе,
- оставить заявку,
- написать в Telegram-бота для быстрой передачи данных.

## 2) Ключевая бизнес-логика (обязательная)
1. Все цены на сайте — **только за работу**.
2. Запчасти / жидкости / расходники — **отдельно**, с помощью в подборе/выкупе.
3. Любая заявка должна сохраняться в БД (`Lead`) и отправляться в Telegram (если есть ENV).
4. Статусы заявок в Telegram-кнопках должны отражаться в БД.
5. `/admin` защищён сессией (cookie), `/admin/login` публичный.

## 3) UX и порядок секций
Текущий зафиксированный порядок ключевых секций:
1. Наши услуги
2. Оставить заявку
3. Калькулятор
4. FAQ

Навигация в хедере должна соответствовать этому порядку.

## 4) Telegram-каналы заявок
Поддерживаются 2 канала входа:
- Web-форма `POST /api/lead`
- Telegram-бот через webhook `POST /api/telegram/callback`
  - `/start` даёт инструкцию
  - следующее текстовое сообщение создаёт `Lead` c `source=tg_bot`
  - заявка пересылается в основной чат `TELEGRAM_CHAT_ID` в формате NeGaraj

## 5) Нельзя ломать без явного апрува
- Формулировку про цены «только за работу».
- Схему БД `Lead`/`Event` и рабочие API-маршруты:
  - `/api/lead`
  - `/api/track`
  - `/api/telegram/callback`
  - `/api/admin/*`
- Порядок секций/навигации и базовую тёмную неоновую стилистику.

## 6) Минимальный definition of done
Фича считается готовой, если:
- нет регрессии в отправке заявок (web + tg bot flow),
- нет циклов редиректов на `/admin`,
- API возвращает ожидаемые JSON/CSV ответы,
- документация остаётся актуальной.
